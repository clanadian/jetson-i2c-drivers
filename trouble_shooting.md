# 트러블슈팅 노트

> 프로젝트 전 과정에서 만난 25건 중, 이 하드웨어(Jetson Nano)·이 커널(L4T 4.9)에서 직접 당해봐야 아는 핵심 5건만 남김.
> 표준 문서에서 쉽게 다시 찾을 수 있는 일반론은 제외.

## 1. probe가 조용히 호출되지 않음 — 구형 커널(4.9)은 `id_table` 필수

**증상**: `insmod` 성공, 드라이버 등록됨, DT compatible 일치, `of_node` 연결 정상, `modinfo`에 OF alias까지 완벽 — 그런데 `probe()`가 호출조차 안 됨. 수동 `bind`도 `No such device`. 로그 한 줄 없음.

**원인**: 4.9 `i2c-core.c`의 probe 래퍼가 매칭 성공 여부와 무관하게 `id_table`을 요구함. 최신 커널엔 없는 조건이라 최신 문서/예제만 보면 알 수 없음.

```c
/* 4.9 i2c_device_probe() */
if (!driver->probe || !driver->id_table)
    return -ENODEV;   /* OF 매칭을 통과해도 id_table 없으면 여기서 끝 */
```

**해결**: `i2c_device_id` 테이블 추가 + `.id_table` 지정. 내용물은 안 쓰이고 "NULL이 아님"만 중요.

**교훈**: "매칭"과 "probe 호출"은 별개 단계. 매칭 재료가 전부 정상인데 probe가 안 불리면 추측하지 말고 그 커널 버전의 버스 코어 probe 래퍼 소스를 직접 읽을 것. 구형 커널에선 레퍼런스 코드가 아니라 타겟 커널 소스가 유일한 진실. (콜백 시그니처도 마찬가지 — 4.9는 probe 2인자, remove int 리턴.)

## 2. EEPROM ACK 폴링이 항상 실패 — Tegra는 zero-length write가 하드웨어적으로 불가

**증상**: write cycle 대기용 ACK 폴링(`i2c_master_send(client, NULL, 0)`)이 매번 `EREMOTEIO`. 타임아웃을 4배 늘려도 성공률 불변. 정작 데이터는 매번 정상 저장돼 있었음(전송 성공, 완료 확인만 실패).

**원인**: Tegra I2C는 패킷 기반 엔진이라 페이로드 길이를 "N-1"로 인코딩 — 0바이트를 표현할 값 자체가 없음. `len == 0`이면 언더플로우로 컨트롤러 에러.

```c
/* L4T 4.9 i2c-tegra.c */
packet_header[1] = msg->len - 1;   /* len==0 → 0xFFFFFFFF */
```

메인라인은 이를 `I2C_AQ_NO_ZERO_LEN` quirk로 공식화해 `-EOPNOTSUPP`로 사전 거절하지만, 4.9엔 미백포트라 애매한 `EREMOTEIO`로 죽어 원인 파악이 오래 걸림. 라즈베리파이 등 지원 컨트롤러에선 같은 코드가 잘 동작 — **Tegra 실리콘 한정** 제약.

**해결**: `msleep(20)` 고정 딜레이로 롤백 (tWR 최대 5ms + 마진). 이 플랫폼에선 임시방편이 아니라 올바른 구현.

**교훈**: 타임아웃을 몇 배 늘려도 성공률이 그대로면 타이밍이 아니라 구조 문제. "표준 기법"도 컨트롤러가 그 트랜잭션 모양을 만들 수 있어야 성립 — 해당 SoC 버스 드라이버의 전송 경로 소스를 직접 읽는 게 제일 빠르고, 메인라인과의 diff(quirk 선언)가 하드웨어 제약 문서 역할을 함.

## 3. SMBus 래퍼가 안 먹히면 `i2c_transfer()` + `i2c_msg` 직접 구성으로 내려갈 것

**증상**: 세 번 당함 — ① `i2c_smbus_read_block_data()`는 SMBus 전용(슬레이브가 길이를 먼저 보냄)이라 일반 I2C 기기에 쓰면 오동작(일반용은 `..._read_i2c_block_data()`). ② AT24C256은 내부 주소가 2바이트라 "command 1바이트" 형식인 래퍼로 표현 불가. ③ 형식이 맞는 MPU6050 14바이트 버스트조차 이 버스에선 래퍼가 매번 `EREMOTEIO` — 같은 시각 같은 버스의 `i2c_transfer()` 경로는 정상이라는 비대칭이 힌트.

**해결**: 주소 write + repeated START read를 `i2c_msg[2]`로 직접 구성. 연속 레지스터는 한 트랜잭션 버스트로 읽어 샘플 시점 정합성 유지.

```c
struct i2c_msg msgs[2] = {
    { .addr = a, .flags = 0,        .len = 1,  .buf = &reg },
    { .addr = a, .flags = I2C_M_RD, .len = 14, .buf = raw },
};
i2c_transfer(client->adapter, msgs, 2);
```

**교훈**: `i2c_smbus_*`는 `i2c_transfer()` 위의 편의 래퍼일 뿐 — 프로토콜 모양이 안 맞거나 래퍼 경로가 의심되면 한 단계 내려갈 것. 같은 버스에 "되는 드라이버"와 "안 되는 드라이버"가 공존하면 두 전송 경로 차이부터 비교하는 게 가장 빠른 진단. (단, 코드와 배선을 동시에 바꾸면 인과 확정 불가 — 변인은 하나씩.)

## 4. 디버깅 방법론 — 계측 먼저, 진단 도구의 사각지대, 버스 진단 규칙

**증상**: 실패 경로에 로그가 없어 "호출 안 됨"과 "호출됐는데 실패"조차 구분 불가. dmesg는 USB 노이즈에 밀려 유실. 죽은 칩이 `i2cdetect`에선 멀쩡해 보임.

**해결/규칙**:
- **계측 먼저**: 커널은 모든 실패 지점에 `pr_err`(원인+에러코드)+성공 시 `pr_info`, 앱은 `errno`/`strerror` 출력. 이거 없인 디버깅 시작 자체가 안 됨.
- **dmesg 유실 대책**: `grep` 필터 < `journalctl -k`(별도 저장) < `sudo dmesg -w > log &` 실시간 캡처 후 재현(가장 확실).
- **`i2cdetect`의 `UU`**: "드라이버 바인딩됨" 표시일 뿐 칩 생존 증명이 아님(바인딩된 주소는 프로브 생략). 칩 확인은 `rmmod` 후 스캔.
- **`readlink -f`**: 없는 경로도 조립해서 출력함. 심볼릭 링크 실재 확인은 `ls -la`.
- **버스 진단 규칙**: I2C 버스는 한 몸 — 깨진 트랜잭션이 SDA를 잡으면 다른 주소까지 전부 죽어 보임(bus hang). **한 기기만 사라짐 = 그 기기 배선/전원, 여러 기기 동시 사라짐 = 공유 자원(버스 행/전원 레일).** 브레드보드에선 일시적 실패를 전제하고 재시도/스킵 방어를 기본 탑재할 것.

**교훈**: 문제가 생기면 가설보다 계측이 먼저. 진단 도구도 각자 사각지대가 있으니 출력의 정확한 의미를 알고 쓸 것.

## 5. 센서는 가정하지 말고 실측으로 검증 — WHO_AM_I, DLPF, 1g 검산

**증상/해결 3종**:
- **칩 ID**: WHO_AM_I를 데이터시트 값(0x68)으로 하드코딩 → probe 실패. 실측(`i2cget -y 1 0x68 0x75`)하니 `0x70`(MPU6500 계열/클론). 저가 브레이크아웃 보드는 실칩이 다를 수 있고, I2C 주소와 WHO_AM_I 값은 별개.
- **노이즈**: raw 값이 수백 LSB씩 출렁인 주범은 배선이 아니라 **DLPF(칩 내장 필터)가 꺼진 파워온 기본값**(260Hz). `CONFIG(0x1A)=0x03`(44Hz) 한 줄로 해결 — 이 필터는 센서 실리콘 안의 하드웨어라 커널이 계산해주는 게 아님. 앱단 N샘플 평균(√N 감소)은 별도로 누적되는 보조 수단.
- **캘리브레이션 검산**: 저장 전에 정지 상태 accel 평균의 벡터 크기 `√(x²+y²+z²) ≈ 16384(1g)` 확인. 크기는 1g인데 방향이 이상하면 센서 고장이 아니라 **덤프 당시 자세가 기울었던 것**(자이로만 정상인 것도 같은 증거). 실사용 자세 기준으로 캘리브레이션하면 중력 방향이 오프셋에 구워지는(baked-in) "상대값" 방식이 된다는 것만 인지하고 쓸 것.

**교훈**: ID 체크·필터 설정·캘리브레이션 전부 "데이터시트의 가정"이 아니라 "내 보드의 실측"으로 확정할 것. 저장 전 물리 법칙(1g)으로 검산하는 습관이 쓰레기 데이터의 전파를 막음.
